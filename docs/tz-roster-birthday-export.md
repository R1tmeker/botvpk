## 1. Поздравления с днём рождения

### 1.1 Поведение
Бот (`backend/app/bot.py`) ежедневно проверяет участников и поздравляет тех, у кого сегодня ДР.
Поздравляются только люди «из состава» (см. §3.1, основной состав), у кого заполнен `birth_date`.

### 1.2 Куда отправлять
- В общий чат (`birthday_chat_id` из настроек) — поздравление с упоминанием участника.
- Опционально (фаза 2): личное сообщение самому имениннику в Telegram.

### 1.3 Редактируемый текст через админку
- Текст-шаблон хранится в таблице `settings`, ключ `birthday_greeting_template`.
- Поддержка плейсхолдеров: `{name}` — ФИО, `{first_name}` — имя, `{age}` — сколько исполнилось (если можно вычислить).
- Пример по умолчанию: `Поздравляем {name} с днём рождения! Желаем успехов в учёбе и службе.`
- Редактируется в админ-вкладке «Настройки» (см. §1.5).

### 1.4 Backend
- **Новый cron-job** в `backend/app/background.py`:
  `scheduler.add_job(send_birthday_greetings, "cron", hour=H, minute=M, max_instances=1)`,
  час/минута — из настроек (`birthday_time`, дефолт 09:00) и `timezone`.
- Логика: выбрать `users` где `status_code='ACTIVE'`, `role_code` ∈ составе, день/месяц `birth_date` = сегодня.
  Учесть 29 февраля через `leap_policy` (`28` или `march1`) — логика была в старом `bot/services/birthdays.py`,
  перенести алгоритм (только алгоритм, не CSV).
- Отправка через `Bot` в `birthday_chat_id`; если чат не задан — джоб тихо пропускается (warning в лог).
- Идемпотентность в рамках суток: не слать повторно (например, помечать в `notifications` или
  проверять, что сегодня уже отправляли — через таблицу `notifications`/`bot_chats` или in-memory guard на интервал).

### 1.5 Расширить whitelist настроек
В `backend/app/routers/admin/settings.py` (`ALLOWED_SETTING_KEYS`) добавить:
`birthday_enabled`, `birthday_chat_id`, `birthday_time`, `birthday_greeting_template`, `leap_policy`.

### 1.6 Frontend
- В админке новая вкладка **«Настройки»** (только для SUPER_ADMIN, как и endpoint):
  форма редактирования ключей из §1.5 (вкл/выкл поздравлений, chat_id, время, шаблон текста с подсказкой о плейсхолдерах).
- Использовать существующие `GET/PATCH /admin/settings` (`useAdminSettings`/`useUpdateSettings` — добавить хуки).

### 1.7 Критерии приёмки
- [x] Ежедневный cron-job в background.py поздравляет именинников из состава.
- [x] Текст редактируется из вкладки «Настройки» в AdminView (SUPER_ADMIN) без перезапуска.
- [x] 29 февраля обрабатывается по политике из настроек (28-е или 1 марта).
- [x] Дублирование исключено — проверка через таблицу notifications (created_at >= сегодня).
- [ ] Requires: birthday_chat_id задан в настройках и бот состоит в этом чате.

## 2. Выгрузка состава в Excel и CSV

### 2.1 Поведение
Командир/админ может выгрузить состав файлом — в формате **CSV** и **XLSX**.
Выгрузка уважает разграничение ролей: командир отделения — только свой отряд; взвод/админ — все.

### 2.2 Backend
- **Новые endpoint'ы** в `backend/app/routers/admin/users.py` (или `reports.py`):
  - `GET /admin/users/export.csv?squad_id=&status=&role=`
  - `GET /admin/users/export.xlsx?squad_id=&status=&role=`
- Те же фильтры и squad-scoping, что в `GET /admin/users` (командир отделения видит только свой отряд).
- Колонки: ФИО, Отделение, Роль (рус. метка), Статус, @username, Телефон, Дата рождения, Дата привязки.
- Исключать PUBLIC_USER (см. §3.1) — экспортируется состав, не случайные заходы.
- CSV: `csv` из stdlib + `StreamingResponse` (как в `reports.py::export_report`), разделитель `;`, BOM для Excel-кириллицы.
- XLSX: добавить зависимость **`openpyxl`** в `backend/requirements.txt` (сейчас её нет).
  Заголовок жирным, автоширина, лист «Состав».
- Логировать выгрузку в audit (`action_code="users.export"`).

### 2.3 Frontend
- В админ-вкладке «Люди» — две кнопки «Выгрузить CSV» / «Выгрузить Excel», уважают текущие фильтры
  (отделение/поиск). Скачивание через blob (паттерн `useExportReport`/`useDownloadFile` уже есть в `queries.ts`).

### 2.4 Критерии приёмки
- [x] CSV открывается в Excel без кракозябр (кириллица — BOM добавлен).
- [x] XLSX открывается, колонки читаемы, роли — русскими метками, заголовок синий.
- [x] Командир отделения выгружает только свой отряд.
- [x] PUBLIC_USER в выгрузку не попадают.

## 3. Раздельные списки состава (вместо «толкучки»)

### 3.1 Логика категорий (обоснование)
`/auth/telegram` (`routers/auth.py:68`) создаёт строку в `users` для каждого, кто открыл приложение,
с `role_code=PUBLIC_USER`. Поэтому общий список замусорен случайными заходами. Раскладка:

| Категория | Критерий | Показывать в составе? |
|---|---|---|
| **Основной состав** | `role_code` ∈ {PARTICIPANT, DEPUTY_SQUAD_COMMANDER, SQUAD_COMMANDER, DEPUTY_PLATOON_COMMANDER, PLATOON_COMMANDER, ADMIN, SUPER_ADMIN}, `status=ACTIVE` | да, основной |
| **Не привязаны** | `role_code=USER_PENDING` (есть в реестре, TG не привязан) | да, подсписок |
| **Заявки (новые)** | `role_code=CANDIDATE` или активная `JoinApplication` | да, отдельный раздел |
| **Случайные заходы** | `role_code=PUBLIC_USER`, нет заявки | нет — исключаем |
| **Архив** | `status=ARCHIVED` | нет по умолчанию, доступен фильтром |

Вывод: твоя логика верна — PUBLIC_USER без заявки в списке состава не нужны.
Их не удаляем из БД (могут позже подать заявку), но в ростере не показываем.

### 3.2 Группировка и вид
- Состав отображается **списком, сгруппированным по отделениям** (заголовок отделения + его участники),
  а не одной плоской кучей.
- Внутри отделения сортировка: командир → зам → участники → не привязанные, далее по алфавиту.
- Переключатель разделов (вкладки/сегменты): **Состав | Заявки | Архив**.
  «Состав» — основной + не привязанные, сгруппированы по отделениям.
  «Заявки» — CANDIDATE / активные заявки (уже есть во вкладке «Заявки» админки — переиспользовать).

### 3.3 Backend
- `GET /admin/users` дополнить:
  - параметр `exclude_public=true` (дефолт для ростера) — отсекает `PUBLIC_USER` без заявки;
  - по умолчанию скрывать `ARCHIVED` (показывать по `status=ARCHIVED`);
  - в ответе уже есть `squad_id` — группировку делает фронт.
- Альтернатива: отдельный endpoint `GET /admin/roster` со структурой `{ squads: [{squad, members[]}], unassigned: [] }`.
  Рекомендуется параметр на существующем endpoint — меньше дублирования.

### 3.4 Frontend
- Вкладка «Люди» в `AdminView` (и/или `PeopleView` для командиров) переделать:
  - сегменты **Состав / Заявки / Архив**;
  - в «Составе» — группировка по отделениям (использовать `squads` + `squad_id`), отдельный блок «Без отделения / Не привязаны»;
  - поиск (уже есть) фильтрует внутри групп;
  - PUBLIC_USER не показываются.
- `PeopleView` (член-facing, уровень 3+) — тот же принцип группировки по отделениям, read-only.

### 3.5 Критерии приёмки
- [x] Состав отображается по отделениям, а не одной кучей.
- [x] PUBLIC_USER без заявки не видны в составе.
- [x] Есть разделы Состав / Заявки / Архив.
- [x] Командир отделения видит только свой отряд (squad-scoping из сессии 6 сохраняется).

## 4. Порядок реализации и оценка

1. §3 (раздельные списки) — самый заметный UX-эффект, чисто фронт + 1 параметр на бэке. ~1–1.5 дня.
2. §2 (выгрузка) — бэкенд endpoint + openpyxl + 2 кнопки. ~1 день.
3. §1 (ДР) — джоб + настройки + вкладка «Настройки». ~1.5–2 дня.

Итого ~3.5–4.5 дня.

## 5. Зависимости и риски
- Добавляется `openpyxl` в requirements → пересборка образа backend.
- Для ДР нужен реально заданный `birthday_chat_id` и бот должен состоять в этом чате.
- Группировка по отделениям опирается на корректные `squad_id` (после миграции сессии 6 — заполнены).
