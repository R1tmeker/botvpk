# Вход на сайт по паролю + канал ВКонтакте

Реализовано 2026-06-02. Две связанные фичи поверх существующей Telegram-аутентификации.

## Главный принцип

**Telegram — основной аккаунт (identity).** Пароль для сайта и привязка ВК — это
дополнительные способы доступа к **уже существующему** аккаунту. Зарегистрироваться
«с нуля» на сайте или в ВК нельзя: человек сначала вступает через Telegram, и только
потом задаёт пароль / привязывает ВК. Одна запись `User` ↔ `telegram_id` + `vk_id` + пароль.

## Фича 1 — Вход на сайт по паролю

### Потоки

**Установка пароля (только в Mini App):**
Профиль → «Вход на сайте» → «Задать пароль» (≥8 символов). Доступно подтверждённым
участникам (роль ≥ PARTICIPANT).

**Вход на сайте (браузер вне Telegram):**
Открывается экран входа → `Telegram ID` + пароль → выдаётся тот же JWT, что и в
Telegram. Сессия хранится в `localStorage` и переживает перезагрузку. Выход — кнопка
«Выйти» в профиле.

**Забыл пароль:** зайти снова через Telegram Mini App и задать новый.

### Эндпоинты (`backend/app/routers/auth.py`)

| Метод | Путь | Доступ | Назначение |
|-------|------|--------|-----------|
| `GET` | `/api/auth/password/status` | Telegram-сессия | Установлен ли пароль |
| `POST` | `/api/auth/password/set` | роль ≥ PARTICIPANT | Задать/сменить пароль (смена требует текущий) |
| `DELETE` | `/api/auth/password` | Telegram-сессия | Отключить вход по паролю |
| `POST` | `/api/auth/password/login` | без токена, **5/min** | Вход: `telegram_id` + `password` → JWT |

### Безопасность
- Argon2-хеш (`backend/app/utils/password.py`, библиотека `pwdlib[argon2]`). Пароли в открытом виде не хранятся.
- `/auth/password/login` ограничен `5/minute` на IP (`backend/app/ratelimit.py`, общий `limiter`).
- Единая ошибка «Invalid Telegram ID or password» для несуществующего ID и неверного пароля (Telegram ID перечислим).
- Вход разрешён только при `status == ACTIVE` и роли ≥ PARTICIPANT — деактивированный пользователь доступ теряет.
- Все попытки (успех/провал/смена) пишутся в audit log.

### Фронтенд
- `frontend/src/screens/LoginScreen.tsx` — экран входа (показывается, когда нет Telegram `initData`).
- `frontend/src/api/client.ts` — токен в `localStorage` (`getStoredToken`/`clearAccessToken`).
- `App.tsx`: при отсутствии `initData` пытается восстановить сессию через `GET /me`, иначе показывает `LoginScreen`; блок «Вход на сайте» и «Выйти» в `ProfileView`.

## Фича 2 — Канал ВКонтакте

ВК — привязанный вторичный канал. Бот простой: кнопки + ссылка на сайт. Без VK Mini App и без VK OAuth.

### Потоки

Привязать ВК можно **двумя способами** (бот предлагает оба):

**1. По паролю (диалог прямо в ВК):**
1. Непривязанный пишет боту → бот показывает кнопку «🔑 Войти по паролю».
2. Первым сообщением — Telegram ID (логин), вторым — пароль (тот же, что для сайта).
3. Бот проверяет (пароль, роль ≥ PARTICIPANT, статус ACTIVE) и записывает `vk_id`.
4. После успеха бот советует удалить сообщение с паролем. Диалог-состояние живёт 10 минут (in-memory), «отмена» сбрасывает.

**2. По коду (из Mini App):**
1. Профиль в Mini App → «ВКонтакте» → «Привязать» → backend выдаёт 6-значный код (10 минут).
2. Человек отправляет код боту ВК.
3. Бот проверяет код и записывает `vk_id`.

Если в ВК пишет непривязанный человек — бот отдаёт инструкцию по обоим способам и ссылку на Telegram-бота.

> Замечание по безопасности: при входе по паролю пароль попадает в историю переписки ВК — бот предлагает удалить это сообщение. Способ по коду этого недостатка лишён (код одноразовый).

**Использование в ВК (pull):** кнопки `📅 Расписание`, `🔔 Уведомления`, `👤 Профиль`, `🌐 Открыть сайт`.

**Push:** уведомления с `send_to_tg=true` дублируются в ВК привязанным пользователям
(`backend/app/background.py` → `send_pending_vk_notifications`, отдельный `vk_sent_at`).

### Эндпоинты (`backend/app/routers/vk.py`)

| Метод | Путь | Доступ | Назначение |
|-------|------|--------|-----------|
| `GET` | `/api/auth/vk/status` | Telegram-сессия | Привязан ли ВК |
| `POST` | `/api/auth/vk/link-code` | роль ≥ PARTICIPANT | Выдать код привязки |
| `DELETE` | `/api/auth/vk/` | Telegram-сессия | Отвязать ВК |

Сам приём кода идёт **не через HTTP**, а в ВК-боте (`backend/app/vk_bot.py`), который
работает с БД напрямую через `redeem_link_code` (`backend/app/utils/channel_link.py`),
по аналогии с Telegram-ботом.

### Безопасность / надёжность
- Коды одноразовые, живут 10 минут; выдача нового гасит прежние неиспользованные.
- `vk_id` уникален: нельзя привязать один ВК к двум аккаунтам.
- Привязка только для роли ≥ PARTICIPANT.
- Отправка в ВК — `messages.send` через `httpx` (`backend/app/utils/vk.py`).

## База данных (миграции)

| Миграция | Изменения |
|----------|-----------|
| `0006_user_password` | `users.password_hash`, `users.password_set_at` |
| `0007_vk_linking` | `users.vk_id` (уникальный), таблица `channel_link_codes`, `notifications.vk_sent_at` |

Применить: `alembic upgrade head` (в контейнере backend или локально из `backend/`).

## Переменные окружения (`.env`)

```
# ВК (по умолчанию выключено — фича 1 работает без этого)
VK_BOT_ENABLED=false
VK_GROUP_TOKEN=          # токен сообщества ВК (Long Poll API)
VK_GROUP_ID=
VK_BOT_URL=              # https://vk.me/your_group — кнопка "Открыть бота ВК"
SITE_URL=               # публичный URL сайта (fallback на MINI_APP_URL)
```

Пароль (фича 1) дополнительных переменных не требует — использует существующий `JWT_SECRET`.

## Деплой

- **Прод (docker-compose.prod.yml):** добавлен сервис `vk_bot` (`python -m app.vk_bot`),
  рядом с `bot`. При `VK_BOT_ENABLED=false` контейнер простаивает (idle), не уходит в restart-loop.
- **systemd:** добавлен `systemd/botvpk-vk.service` (альтернативный путь запуска).
- Зависимости: `pwdlib[argon2]` и `vkbottle` добавлены в `requirements.txt` и `backend/requirements.txt`.

Порядок выката: применить миграции → пересобрать backend/frontend → (опционально) включить `VK_BOT_ENABLED` и задать токены.

## Что НЕ входит (план на будущее)
- Сброс пароля через бота (сейчас — повторно через Mini App).
- Вход «Через ВК» (VK OAuth) на сайте — сайт остаётся парольным.
- Вход по username/телефону, 2FA, refresh-токены.
