# ТЗ: аудит безопасности, доработка админки и унификация ботов

> Консолидированное техническое задание по итогам аудита (май 2026).
> Связанные документы: `tz-implementation-roadmap.md`, `bot-db-unification-strategy.md`.
> Статус разделов: **[СДЕЛАНО]** — реализовано в коде; **[К ВЫПОЛНЕНИЮ]** — план; **[ВНЕ ОБЪЁМА]** — отложено по решению заказчика.

---

## 0. Контекст и цели

Информационная система ВПК «Звезда» (Telegram Mini App + бот + FastAPI backend + PostgreSQL).
Аудит выявил три блока проблем: уязвимости разграничения ролей (RBAC), неполнота админки,
архитектурный долг — сосуществование двух ботов и legacy CSV-хранилища.

Цели ТЗ:
1. Закрыть критические уязвимости и дыры RBAC на бэкенде.
2. Довести админ-панель до рабочего состояния (все сущности управляются из UI).
3. Свести систему к единому боту (`backend/app/bot.py`) на общей PostgreSQL, мигрировать
   реальные данные участников из CSV, вывести legacy-бота из эксплуатации.

## 1. Безопасность и RBAC (backend) — [СДЕЛАНО]

Реализовано в сессии аудита. Зафиксировано здесь как часть объёма.

### 1.1 Критические уязвимости
- **Privilege escalation при приёме заявки** (`routers/admin/applications.py`): `role_code`
  ограничен whitelist'ом (`PARTICIPANT`, `DEPUTY_SQUAD_COMMANDER`, `SQUAD_COMMANDER`);
  `squad_id` проверяется на существование и активность.
- **IDOR на файлах** (`routers/files.py`): доступ к `GET /files/{id}` и `/files/{id}/download`
  только загрузчику файла или роли ≥ DEPUTY_PLATOON_COMMANDER; скачивания логируются в audit.
- **Утечка чужих данных в экспорте** (`routers/reports.py`): `/reports/export` принимает
  `squad_id` и передаёт его в подотчёты (раньше игнорировал → выгружались все отделения).
- **Произвольные ключи настроек** (`routers/admin/settings.py`): whitelist `ALLOWED_SETTING_KEYS`,
  неизвестные ключи → 400.

### 1.2 Целостность RBAC
- Проверка `can_manage_squad()` на PATCH/generate шаблонов расписания (`routers/schedule.py`).
- Squad-ownership и запрет повышения роли выше своей при PATCH пользователя (`routers/users.py`).
- `GET /admin/users` доступен с уровня DEPUTY_SQUAD_COMMANDER; командиры отделений видят
  только свой отряд; добавлен параметр `?search=`. Добавлены `PATCH /admin/users/{id}` и
  `PATCH /admin/users/{id}/deactivate` (`routers/admin/users.py`).
- Фильтры аудит-лога: `user_id`, `action_code`, `entity_name`, пагинация `offset`
  (`routers/admin/audit.py`).
- Стандартизированы права promo: DELETE снижен до DEPUTY_PLATOON_COMMANDER
  (`routers/admin/promo.py`).

### 1.3 Остаточный долг безопасности — [К ВЫПОЛНЕНИЮ, низкий приоритет]
- `audit_log.ip_address` не заполняется — прокинуть `Request` в endpoint'ы.
- Опционально: CHECK-constraint / Enum на `users.role_code` и `status_code` (сейчас свободные строки).
- Инвалидация JWT при понижении роли (частично закрыто — роль берётся из свежего объекта в `auth.py`).

## 2. Админ-панель (frontend `screens/App.tsx`) — [СДЕЛАНО]

Реализовано в сессии аудита.
- Новые вкладки: **Обращения**, **События кандидатов**, **Нормативы**, **Материалы**.
- Поиск по пользователям; фильтр заявок по статусу; промежуточные статусы заявок
  (REVIEWING, NEEDS_INFO, INVITED_MEETING и т.д.).
- Управление обращениями (смена статуса).
- Логи с фильтрами по `action_code`/`entity_name`, показ `new_value`.
- Подтверждение `confirm()` перед удалением (события, промо, нормативы, деактивация).
- Доступ командира отделения (уровень 4) к вкладке «Люди».
- Меню: сортировка вверх/вниз через `sort_order`, снято ограничение на 12 карточек.
- Деактивация пользователей (уровень 8+).
- Новые API-хуки в `api/queries.ts` (см. историю изменений в memory проекта).

### 2.1 Сверка справочников — [К ВЫПОЛНЕНИЮ]
В новых вкладках захардкожены коды статусов/аудиторий, требующие сверки с моделями:
- статусы обращений (`IN_PROGRESS / NEEDS_INFO / RESOLVED / CLOSED`) — сверить с `models/appeal.py`;
- audience-коды материалов/курсов (`MEMBER / CANDIDATE / ALL`) — сверить с `models/learning.py`.
- Технический долг: дублирование ручек редактирования пользователя
  (`/users/{id}` и `/admin/users/{id}`) — позже свести к одной.

## 3. Унификация ботов — [К ВЫПОЛНЕНИЮ]

### 3.1 Исходная ситуация
В репозитории два бота:
- **`backend/app/bot.py`** — DB-first (PostgreSQL), разворачивается в проде (docker-compose сервис
  `bot`: `python -m app.bot`). Единственный источник истины. Рассинхрона ролей не имеет.
- **`bot/` + корневой `main.py`** — legacy CSV-бот, не в docker-compose. Подлежит удалению.

### 3.2 Решения заказчика (зафиксированы)
- **Опросы (polls)** — НЕ используются в групповом чате → **[ВНЕ ОБЪЁМА]**, не портируются, удаляются вместе с legacy.
- **Поздравления с ДР** — **[ВНЕ ОБЪЁМА]**, не портируются в этой итерации.
  Важно: после удаления legacy автопоздравлений не будет. При необходимости —
  оформить отдельной задачей позже.
- **Данные участников из CSV** — мигрировать все 27 записей в таблицу `users`.

### 3.3 Объём работ
1. Подтвердить, что в проде запущен только DB-бот (один polling на токен — иначе 409 Conflict).
2. Мигрировать данные `data/members.csv` → `users` (см. раздел 4).
3. Удалить пакет `bot/`, корневой `main.py`, `config.txt`/`config.example.txt`, `data/*.csv`,
   а также `bot/utils/roles.py` (legacy-модель ролей). Останется единственная модель
   `backend/app/roles.py`.
4. Подчистить `requirements`/Docker от зависимостей, нужных только CSV-боту (если есть).
5. Обновить `tz-implementation-roadmap.md`: DB-first бот — единственный.

### 3.4 Маппинг legacy-ролей при миграции
| CSV-роль | role_code в БД |
|---|---|
| SUPER_ADMIN | PLATOON_COMMANDER |
| ADMIN | DEPUTY_PLATOON_COMMANDER |
| LEAD | SQUAD_COMMANDER |
| USER_CONFIRMED | PARTICIPANT |
| USER_PENDING | USER_PENDING |
| остальные | 1:1 |

Статусы: `active → ACTIVE`, `removed → ARCHIVED`.

Обоснование безопасности: реальный супер-админ определяется в `auth.py` по
`telegram_id == settings.super_admin_id`, поэтому маппинг `SUPER_ADMIN → PLATOON_COMMANDER`
не лишает владельца прав и не выдаёт лишних.

## 4. Подготовка БД и миграция данных — [К ВЫПОЛНЕНИЮ]

Скрипт миграции уже написан и проверен (компиляция + импорт): `backend/app/scripts/migrate_csv_members.py`.
Idempotent: upsert по `telegram_id`, для непривязанных — по `(full_name, birth_date)`.

### 4.1 Порядок выполнения (в окружении с доступом к PostgreSQL)
1. **Бэкап БД** (обязательно):
   `pg_dump` или взять существующий дамп `backups/vpk-zvezda-*.dump` как точку отката.
2. **Применить миграции схемы** (если есть незакоммиченные): `alembic -c alembic.ini upgrade head`.
3. **Сухой прогон** миграции данных (ничего не пишет, печатает отчёт):
   `python -m app.scripts.migrate_csv_members --csv ../data/members.csv`
4. **Сверить отчёт**: сколько создать/обновить/без изменений, какие новые отделения,
   расхождения ролей. Убедиться, что цифры ожидаемы (≈27 участников, отделения «1/2/3 Отделение»).
5. **Применить**: `python -m app.scripts.migrate_csv_members --csv ../data/members.csv --apply`
6. **Постпроверка** (раздел 4.2).

### 4.2 Критерии корректности миграции
- В `users` присутствуют все активные участники из CSV (по `telegram_id` для привязанных).
- Нет дублей (повторный прогон скрипта показывает «без изменений»).
- В `users.role_code` отсутствуют legacy-коды (`LEAD`, `USER_CONFIRMED`, `SUPER_ADMIN` и т.п.
  на тех, кто реально не техадмин).
- Отделения из CSV («1 Отделение», «2 Отделение», «3 Отделение») существуют в `squads`.
- Реальный владелец (`r1tmeker`, tg 795307805) получает SUPER_ADMIN через `auth.py`.

### 4.3 Откат
При сбое — восстановить БД из бэкапа п.4.1. Скрипт идемпотентен, повторный прогон безопасен.
