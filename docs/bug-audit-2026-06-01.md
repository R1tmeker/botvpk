### BUG-01 · Утренняя сводка пропускает ранние занятия

**Файл:** `backend/app/background.py:325–340`

**Описание:** Планировщик настроен на `Asia/Novosibirsk` (UTC+7), но вычисление диапазона дня производится в UTC:

```python
now = utcnow()  # 00:00 UTC = 07:00 NSK
today_start = now.replace(hour=0, minute=0, second=0, microsecond=0)  # UTC midnight
today_end = today_start + timedelta(days=1)
```

Занятие, запланированное на **04:00 NSK = 21:00 UTC вчера**, не попадает в диапазон `[00:00 UTC сегодня, 00:00 UTC завтра)`. Пользователи не получат утреннее уведомление об этом занятии.

**Фикс:**
```python
import pytz
tz = pytz.timezone(settings.timezone)
now_local = datetime.now(tz)
today_local_start = now_local.replace(hour=0, minute=0, second=0, microsecond=0)
today_start = today_local_start.astimezone(timezone.utc).replace(tzinfo=None)
today_end = (today_local_start + timedelta(days=1)).astimezone(timezone.utc).replace(tzinfo=None)
```

### BUG-02 · Кнопка "Изменить ответ" COMING → NOT_COMING обходит выбор причины

**Файл:** `frontend/src/screens/App.tsx:1182–1193`

**Описание:** Когда пользователь уже ответил "Приду" (COMING) и нажимает "Изменить ответ", приложение отправляет `NOT_COMING` напрямую без запроса причины отсутствия. Backend отклоняет запрос ошибкой 400 (`"Absence reason is required."`), но UI не показывает ошибку явно.

```tsx
// Было — переключает ответ без учёта requiresResponse
onRespond(eventId, currentResponse === "COMING" ? "NOT_COMING" : "COMING");
```

**Фикс:**
```tsx
onClick={() => {
  const newCode = currentResponse === "COMING" ? "NOT_COMING" : "COMING";
  if (newCode === "NOT_COMING" && requiresResponse) {
    setPickingReason(true);
  } else {
    onRespond(eventId, newCode);
  }
}}
```

### BUG-03 · Аудит-запись нового норматива через бот имеет entity_id = null

**Файл:** `backend/app/bot.py:968–977`

**Описание:** При первой сдаче норматива через бот (новый `NormativeSubmission`), объект добавляется через `session.add()` без `await session.flush()`. Вызов `record_audit` с `entity_id=getattr(existing, "id", 0)` возвращает `None` до commit. Аудит-лог получает `entity_id=None`.

```python
if existing is None:
    existing = NormativeSubmission(...)
    session.add(existing)
await record_audit(
    ...
    entity_id=getattr(existing, "id", 0),  # None до flush!
)
await session.commit()
```

**Фикс:**
```python
await session.flush()  # Присваивает existing.id
await record_audit(
    ...
    entity_id=existing.id,
)
await session.commit()
```

### BUG-04 · `today_schedule` использует naive datetime для timezone-aware колонок

**Файл:** `backend/app/routers/schedule.py:113–121`

**Описание:** Endpoint `/schedule/today` создаёт naive datetime объекты и сравнивает с timezone-aware колонками в PostgreSQL. Это приводит к ошибке `"can't compare offset-naive and offset-aware datetimes"` или молчаливому неверному результату в зависимости от драйвера.

```python
today = date.today()
start = datetime.combine(today, time.min)   # naive!
end = datetime.combine(today, time.max)     # naive!
return await list_schedule(start, end, None, current_user, session)
```

**Фикс:**
```python
today = date.today()
start = datetime.combine(today, time.min, tzinfo=timezone.utc)
end = datetime.combine(today, time.max, tzinfo=timezone.utc)
```

### BUG-05 · `datetime.utcnow()` — deprecated, возвращает naive datetime

**Файл:** `backend/app/routers/users.py:59, 101`

**Описание:** Используется устаревший `datetime.utcnow()` (deprecated с Python 3.12). Возвращает naive datetime, тогда как все остальные поля `DateTime(timezone=True)` хранят timezone-aware значения. Вызывает несогласованность и предупреждения на Python 3.12+.

```python
user = User(**payload.model_dump(), updated_at=datetime.utcnow())  # строка 59
user.updated_at = datetime.utcnow()  # строка 101
```

**Фикс:** Заменить на `datetime.now(timezone.utc)` (как во всех остальных роутерах).

### BUG-07 · Двойной commit в `reject_application` — риск потери уведомления

**Файл:** `backend/app/routers/admin/applications.py:193–210`

**Описание:** `reject_application` вызывает `update_application` (который делает `session.commit()`), а затем добавляет уведомление и делает второй `session.commit()`. Если второй commit упадёт — заявка уже в статусе REJECTED, но кандидат не получит уведомление.

```python
async def reject_application(...):
    application = await update_application(...)  # commit #1 внутри
    user = await session.scalar(...)
    if user:
        session.add(Notification(...))
        await session.commit()  # commit #2
    return application
```

**Фикс:** Объединить всю логику в одну транзакцию без промежуточного commit.
