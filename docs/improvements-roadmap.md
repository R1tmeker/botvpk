# Часть 1 — Взаимодействие с ботами

Здесь — **только Telegram-бот и VK-бот**, без мини-аппа. Сначала фиксируем, как всё работает сейчас, затем — проблемы и желаемые сценарии.

## 1.1 Telegram-бот — что есть сейчас

Бот построен на aiogram (`backend/app/bot.py`), запускается отдельным процессом (`python -m app.bot`), хранит состояние диалогов в памяти (`MemoryStorage`).

### Команды (зарегистрированы в меню Telegram)
| Команда | Кто | Что делает |
|---------|-----|-----------|
| `/start`, `/menu` | все | Приветствие, роль, reply-клавиатура. Поддерживает deep-link `?start=viewN` — командир сразу получает файл сдачи норматива №N |
| `/profile` | все | ФИО, роль, отделение, статус |
| `/join` | гость/кандидат | Пошаговая анкета вступления (FSM): ФИО → дата рождения → телефон → мотивация → источник → подтверждение |
| `/schedule` | участник+ | Ближайшие 7 событий; если 2+ без ответа — предлагает «ответить на все разом» |
| `/notifications` | участник+ | Последние 10 уведомлений |
| `/attendance` | участник+ | Личные отметки посещаемости |
| `/normatives` | участник+ | Активные нормативы с дедлайнами |
| `/export` | зам. командира отд.+ | Выгрузка состава в CSV / Excel |
| `/admin` | админ | Шпаргалка по резервным командам |
| `/broadcast <текст>` | админ | Рассылка всем активным |
| `/help` | все | Список команд |

### Reply-клавиатура (кнопки снизу)
«Расписание», «Уведомления», «Заявки» (для командиров), «Открыть приложение» (web_app).

### Текстовые триггеры (без слэша)
«Расписание», «Уведомления», «Заявки», «Ссылка туннеля» (служебная).

### Инлайн-кнопки (callback)
- **Ответ на событие**: «Приду» / «Не приду» / «Уточню» (`event:<id>:<code>`)
- **Причина отсутствия**: выбор из справочника, при необходимости — ввод текстом (`reason:`)
- **Пакетный ответ**: «На все» / «Ни на одно» / «По одному» (`batch:`)
- **Проверка норматива командиром**: «Принять» / «Доработка» / «Отклонить» (`norm_review:`)
- **Сдача файла**: выбор норматива, к которому относится присланный файл (`norm_submit:`)
- **Экспорт**: формат CSV / Excel (`export:`)

### Особые сценарии
- **Сдача норматива файлом**: участник просто присылает видео/фото/документ → бот спрашивает, к какому нормативу это относится → сохраняет сдачу и уведомляет командиров.
- **Инлайн-режим**: `@бот` в любом чате показывает ближайшие события (можно делиться расписанием).
- **Заявки для командиров**: кнопка «Заявки» показывает активные заявки на вступление.

## 1.2 Telegram-бот — проблемы

1. **Нет `/cancel`.** Если человек начал `/join` (анкету) или ввод причины отсутствия, выйти штатно нельзя — только дослать до конца или бросить «висящее» состояние. Это частый тупик для новичков.
2. **Рассылка без троттлинга** (`broadcast`, цикл `send_message` без пауз). Telegram режет ~30 сообщений/с; на большом составе бот словит `429` и временную блокировку, часть людей не получит сообщение.
3. **Состояние в памяти** (`MemoryStorage`). Рестарт бота/деплой = все незавершённые анкеты и диалоги обнуляются молча.
4. **Нет привязки VK из Telegram.** Код привязки сейчас выдаётся только в мини-аппе. Человеку без мини-аппа некуда взять код.
5. **Нет обращений и материалов через бот.** «Проблему» (обращение командиру) и учебные материалы можно открыть только в мини-аппе.
6. **Командир не может отметить явку через бот** — только в мини-аппе. На выезде/без приложения это неудобно.
7. **Единый обработчик ошибок отсутствует.** Исключение в одном хендлере пишется в лог, но пользователь видит «тишину»; нет дружелюбного «попробуйте позже».
8. **`/help` поверхностный** — не объясняет сценарии (как сдать норматив, как ответить на занятие).
9. **Дубли логики** с мини-аппом и VK-ботом (расписание, нормативы) — при изменении правил легко получить расхождение поведения каналов.

## 1.3 Telegram-бот — желаемые сценарии (что добавить)

### B-TG-1 · `/cancel` и выход из любого диалога · S
Команда `/cancel` (и кнопка «Отмена» в анкете) сбрасывает FSM-состояние на любом шаге. Плюс таймаут «зависшей» анкеты (например, сутки).
**Варианты:** (а) глобальный хендлер `/cancel` — просто и достаточно; (б) к каждому шагу анкеты добавить кнопку «Отмена» — нагляднее, но больше кода. **Рекомендация:** оба — глобальный `/cancel` + кнопка в анкете.

### B-TG-2 · Привязка VK прямо из Telegram · S
Команда `/vk` (или кнопка в меню): бот выдаёт 6-значный код привязки и ссылку на VK-сообщество. Переиспользует `issue_link_code` из backend.
**Эффект:** человек получает код, не открывая мини-апп.

### B-TG-3 · Богатое главное меню на инлайн-кнопках · M
Сейчас разделы раскиданы по reply-кнопкам и командам. Сделать `/start` с инлайн-меню: «📅 Расписание», «🔔 Уведомления», «📋 Нормативы», «📊 Моя статистика», «📨 Обращение», «🔗 Привязать VK», «🌐 Открыть приложение».
**Варианты:** (а) reply-клавиатура — привычно, но занимает место ввода; (б) инлайн под сообщением — компактно, можно перерисовывать; (в) гибрид. **Рекомендация:** инлайн-меню для разделов + одна reply-кнопка «Меню».

### B-TG-4 · Отметка явки командиром через бот · M
После события бот присылает командиру список участников с кнопками статуса (присутствовал / опоздал / отсутствовал) — отметка в пару тапов без мини-аппа.
**Варианты:** (а) пролистываемый список по одному человеку — надёжно на больших отделениях; (б) одно сообщение со всеми + инлайн-кнопки — быстрее, но упирается в лимит кнопок Telegram (~100). **Рекомендация:** список постранично, по 8–10 человек.

### B-TG-5 · Обращения через бот · M
Команда `/appeal` или кнопка: пошагово тема → текст → (опц. срочность) → отправка. Командирам приходит уведомление, ответы — туда же в бот.
**Эффект:** «Проблему» можно отправить и получить ответ, не заходя в мини-апп.

### B-TG-6 · Быстрые действия в уведомлениях · S
К пуш-уведомлениям добавлять контекстные кнопки: у напоминания о занятии — «Приду/Не приду», у нормотива — «Открыть», у обращения — «Ответить». Часть уже есть в `_build_notification_keyboard` — расширить покрытие.

### B-TG-7 · Подробный `/help` со сценариями · S
Не просто список команд, а короткие инструкции: «Чтобы сдать норматив — пришлите видео», «Чтобы ответить на занятие — нажмите кнопку под уведомлением» и т.д.

## 1.4 VK-бот — что есть сейчас

Бот на vkbottle (`backend/app/vk_bot.py`), отдельный сервис (`python -m app.vk_bot`), включается флагом `VK_BOT_ENABLED`. Принцип: **VK — вторичный канал поверх Telegram-аккаунта**, зарегистрироваться с нуля нельзя, только привязать.

### Сценарии
- **Привязка двумя способами:**
  1. **По паролю (диалог)**: кнопка «🔑 Войти по паролю» → первым сообщением Telegram ID, вторым — пароль (тот же, что для сайта). Состояние диалога живёт 10 минут в памяти, «отмена» сбрасывает.
  2. **По коду**: 6-значный код из мини-аппа/Telegram → одно сообщение.
- **Кнопки после привязки**: «📅 Расписание», «🔔 Уведомления», «👤 Профиль», «🌐 Открыть сайт».
- **Push-уведомления**: дублируются в VK привязанным (отдельный флаг `vk_sent_at`).
- **Для непривязанных**: инструкция по обоим способам + ссылка на Telegram-бота.

## 1.5 VK-бот — проблемы и желаемое

1. **Нет паритета с Telegram.** В VK нельзя ответить на занятие (приду/не приду), сдать норматив, написать обращение — только смотреть.
2. **Состояние диалога в памяти** (`_vk_login_state`) — теряется при рестарте и не шарится между репликами.
3. **Нет отвязки через бот** — только в мини-аппе.
4. **Пароль в переписке.** При входе по паролю он остаётся в истории VK (бот советует удалить, но это ручное действие).

### B-VK-1 · Ответы на занятия в VK · M
Кнопки «Приду/Не приду/Уточню» под напоминанием о занятии (VK keyboard + callback). Переиспользует ту же логику ответа, что и Telegram.
**Варианты:** (а) inline-callback кнопки (VK их поддерживает) — как в Telegram; (б) текстовые кнопки клавиатуры — проще, но засоряет диалог. **Рекомендация:** callback-кнопки.

### B-VK-2 · Сдача норматива в VK · M
Приём вложения (видео/фото/документ) → выбор норматива → сдача, как в Telegram. VK отдаёт вложения в `message.attachments`.

### B-VK-3 · Обращения и отвязка в VK · S
Команда/кнопка «Отвязать» и «📨 Обращение» — паритет с мини-аппом.

### B-VK-4 · Состояние диалога в Redis · S
(см. REL-4) — диалог входа переживает рестарт и масштабирование.

## 1.6 Сквозные улучшения для обоих ботов

### B-ALL-1 · Троттлинг рассылок · M
Telegram ≈30 msg/s, VK тоже лимитирован. Сейчас рассылки/пуши идут в цикле без пауз → `429`/бан.
**Варианты:** (а) простая пауза `asyncio.sleep` между отправками — минимально; (б) семафор/лимитер по скорости — точнее; (в) очередь (Redis) с воркером — масштабируемо. **Рекомендация:** начать с (а)+обработки `RetryAfter`, при росте — (в).

### B-ALL-2 · Глобальный обработчик ошибок · S
aiogram `errors`-роутер и vkbottle error-handler: логировать (Sentry), отвечать пользователю «Произошла ошибка, попробуйте позже», не ронять polling.

### B-ALL-3 · Идемпотентность привязки · S
Поймать `IntegrityError` на гонке уникального `vk_id`, выдать понятное сообщение вместо стектрейса.

### B-ALL-4 · Единый сервисный слой · L
(см. BOT-1) — расписание/нормативы/уведомления/ответы вынести в `app/services/`, чтобы Telegram, VK и API звали один код. Сейчас три почти одинаковые реализации.

# Часть 2 — P0 Безопасность

### SEC-1 · Включить проверку возраста Telegram initData · S
**Проблема.** В `backend/app/config.py:23` значение `TELEGRAM_INIT_DATA_MAX_AGE_SECONDS=0`, то есть проверка отключена. Подпись Telegram остаётся валидной **вечно**: если строка `initData` где-то утекла (история браузера, лог, прокси), ею можно логиниться сколько угодно (replay-атака).
**Варианты.**
- *(а) Поставить 86400 (сутки).* Простая правка `.env`. Раз в сутки человек пере-открывает приложение из бота — initData обновляется. **Минус:** очень редкий кейс долгой неактивной вкладки потребует переоткрытия.
- *(б) Короткое окно (1–4 часа) + автorefresh.* Безопаснее, но если фронт не переполучает initData, участятся «перелогины».
- *(в) Оставить 0.* Не рекомендуется.
**Рекомендация:** (а) 86400. Код уже умеет (`telegram_auth.py:77`), нужно лишь значение.
**Влияние на бота:** никакого — это про мини-апп/веб-сессии.

### SEC-2 · Заголовки безопасности в nginx · S
**Проблема.** В `nginx/nginx.prod.conf` есть `X-Frame-Options` и `X-Content-Type-Options`, но нет HSTS, CSP, Referrer-Policy, Permissions-Policy. Это оставляет окна для downgrade-атак, XSS и утечки реферера.
**Варианты.**
- *(а) Базовый набор:* `Strict-Transport-Security`, `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` (запрет камеры/гео/микрофона). Низкий риск сломать что-то.
- *(б) + строгий CSP.* Максимальная защита, но требует аккуратной настройки под Telegram WebApp SDK и инлайн-стили (в проекте много inline-стилей) — иначе сломается вёрстка. Вводить с `Content-Security-Policy-Report-Only` сначала.
**Рекомендация:** (а) сразу, (б) — отдельной итерацией через report-only.

### SEC-3 · Блокировка после серии неудачных паролей · M
**Проблема.** `password_login` ограничен `5/min` на IP. Это не мешает: (1) перебирать 5 паролей в минуту бесконечно, (2) обойти лимит сменой IP (ботнет/прокси). Lockout по аккаунту отсутствует.
**Варианты.**
- *(а) Счётчик неудач на аккаунт + временная блокировка.* Поля `failed_login_count` и `locked_until` в `User`; после N неудач — пауза с экспоненциальным ростом. Просто, надёжно. **Минус:** возможна намеренная «блокировка чужого аккаунта» перебором — смягчается тем, что логин = Telegram ID (не публичен массово) и блокировка короткая.
- *(б) Лимит по (IP + аккаунт) в Redis.* Гибче, не блокирует «жертву» полностью.
- *(в) CAPTCHA после N попыток.* На сайте реально, в боте — нет.
**Рекомендация:** (б) если будет Redis (REL-4), иначе (а). Плюс `URGENT`-уведомление админам при всплеске неудач.
**Влияние на бота:** вход по паролю в VK (B-VK через `_try_password_link`) должен использовать ту же защиту.

### SEC-4 · Политика паролей · S
**Проблема.** `PasswordSetRequest` требует только `min_length=8`. Допустимы `12345678`, пароль = Telegram ID и т.п.
**Варианты.**
- *(а) Простые правила:* запрет только-цифр, запрет = Telegram ID, минимум буквы+цифры.
- *(б) Чёрный список топ-паролей* (встроенный короткий список или `zxcvbn` для оценки силы). `zxcvbn` точнее, но это доп. зависимость и вес.
**Рекомендация:** (а) сразу + короткий чёрный список; `zxcvbn` — опционально.

### SEC-5 · Проверка файлов по содержимому, а не по `Content-Type` · M
**Проблема.** В `routers/files.py` и `auth.py:upload_my_avatar` тип файла берётся из `Content-Type` запроса — заголовок подделывается, можно загрузить исполняемый файл «как картинку».
**Варианты.**
- *(а) Магические байты* (`filetype`/`python-magic`) + белый список расширений. Отсекает большинство подделок.
- *(б) Пере-кодирование картинок через Pillow* (open→save) — гарантированно срезает встроенные payload-ы и EXIF, заодно нормализует формат. Для видео/PDF — только проверка сигнатуры.
- *(в) Антивирус (ClamAV)* как отдельный сервис — максимально, но тяжело для этого масштаба.
**Рекомендация:** (а)+(б). (в) — избыточно.
**Влияние на бота:** сдача нормативов файлами идёт и через ботов — проверку лучше делать в общем сервисном слое (BOT-1), чтобы покрыть все каналы.

### SEC-6 · Ревокация JWT / версия токена · M
**Проблема.** `utils/jwt.py` выдаёт токен на 24ч без возможности отзыва. При компрометации или деактивации участника доступ сохраняется до истечения.
**Варианты.**
- *(а) `token_version` в `User`.* В токен кладём версию, в `get_current_user` сверяем. Смена пароля/деактивация → инкремент → все старые токены мертвы. Дёшево, без Redis. **Минус:** «всё или ничего» (нельзя отозвать одно устройство).
- *(б) Чёрный список `jti` в Redis.* Точечный отзыв конкретного токена. Нужен Redis.
- *(в) Refresh-токены + короткий access (15 мин).* Правильно, но заметно больше работы на фронте и в ботах.
**Рекомендация:** (а) сейчас (закрывает деактивацию/смену пароля), (в) — когда дойдут руки до полноценной сессионной модели.

### SEC-7 · Секреты вне git и ротация · S
**Проблема.** В корне репозитория присутствует реальный `.env`. Нужно убедиться, что он в `.gitignore` и не попал в историю git (`BOT_TOKEN`, `JWT_SECRET`, пароль БД).
**Варианты.**
- *(а) Проверить и вычистить.* `git log` по `.env`; если был закоммичен — отозвать `BOT_TOKEN` у @BotFather, сменить `JWT_SECRET` (инвалидирует сессии), сменить пароль БД.
- *(б) Секрет-менеджер* (Docker secrets / Vault) — для будущего, избыточно сейчас.
**Рекомендация:** (а) обязательно и срочно.

# Часть 3 — P1 Надёжность и инфраструктура

### REL-1 · Автоматические бэкапы БД · M
**Проблема.** В `backups/` лежит один ручной дамп от 2026-05-28. Сбой диска/ошибочный `DROP` = потеря всего состава, посещаемости, истории.
**Варианты.**
- *(а) cron + `pg_dump` локально* с ротацией (7 ежедневных, 4 недельных). Минимум, но бэкап на том же сервере — при потере сервера теряется и он.
- *(б) Локально + выгрузка offsite* (S3-совместимое хранилище / другое облако). Правильный минимум для прод-данных людей.
- *(в) Managed Postgres* с бэкапами провайдера — снимает вопрос, но это смена инфраструктуры.
**Рекомендация:** (б). Обязательно — отдельно протестировать **восстановление** (бэкап без проверенного restore не считается).

### REL-2 · Healthcheck для ботов · S
**Проблема.** В `docker-compose.prod.yml` healthcheck только у `backend`. Если у `bot`/`vk_bot` умрёт long-polling (сетевой сбой, исключение в библиотеке), контейнер останется «зелёным», а бот — немым. Узнаем от людей: «бот не отвечает».
**Варианты.**
- *(а) Heartbeat-файл:* бот раз в N секунд обновляет `mtime` файла; healthcheck проверяет свежесть. Просто.
- *(б) Внутренний `/health` у бота* (мини aiohttp-сервер) — чище, но +порт/код.
- *(в) Внешний watchdog* (uptime-бот пингует тестовым сообщением) — сложнее.
**Рекомендация:** (а).

### REL-3 · Sentry для ошибок (backend + frontend + боты) · M
**Проблема.** Ошибки видно только в логах контейнеров постфактум; нет агрегации, алертов, контекста (кто, что нажал).
**Варианты.**
- *(а) Sentry SDK* во все три процесса (API, bot, vk_bot) + фронт. Богатый трейсинг, алерты. Облако или self-hosted **GlitchTip** (совместим, бесплатный).
- *(б) Только структурные логи + grep* — дёшево, но без алертов и поиска.
**Рекомендация:** (а), GlitchTip если важна приватность данных.

### REL-4 · Redis: состояния диалогов и кэш · M
**Проблема.** aiogram FSM (`MemoryStorage`) и VK-логин (`_vk_login_state`) живут в памяти процесса: рестарт = потеря незавершённых диалогов; масштабирование на 2+ реплики невозможно (состояние не шарится).
**Варианты.**
- *(а) Redis* как FSM-storage (aiogram имеет `RedisStorage`) + хранилище VK-диалогов + кэш справочников/меню. Один новый сервис в compose.
- *(б) Хранить состояние в Postgres.* Без новой инфраструктуры, но БД нагружается мелкими частыми записями.
**Рекомендация:** (а). Заодно разблокирует SEC-3(б) и SEC-6(б).

### REL-5 · Постоянный домен вместо quick-tunnel · M
**Проблема.** `systemd/vpk-tunnel.service` поднимает Cloudflare **quick** tunnel, у которого URL меняется при каждом рестарте; поэтому есть костыль `scripts/notify-miniapp-url.sh`, рассылающий новый адрес. Это хрупко: между рестартом и рассылкой Mini App/сайт недоступны по старому адресу.
**Варианты.**
- *(а) Именованный Cloudflare tunnel* (привязан к домену, URL постоянный). Бесплатно, минимальная миграция.
- *(б) Свой домен + nginx + Let's Encrypt* (certbot уже есть в compose). Полный контроль, нужен домен и DNS.
**Рекомендация:** (а) как быстрый шаг, (б) — целевое состояние.

### REL-6 · Привести в порядок цепочку миграций · S
**Проблема.** В `backend/alembic/versions/` два файла с префиксом `0002` (`0002_alternating_schedule`, `0002_user_avatar`). Сейчас работает (линейно через `down_revision`), но нумерация вводит в заблуждение и провоцирует ошибки при ручных правках.
**Варианты.**
- *(а) Переименовать по фактическому порядку* и задокументировать. Безопасно (ревизии-идентификаторы внутри файлов важнее имён).
- *(б) Оставить как есть, добавить `alembic check`/`heads` в CI* — чтобы хотя бы ловить расхождения.
**Рекомендация:** (а)+(б).

### REL-7 · Троттлинг рассылок · M
То же, что [B-ALL-1](#b-all-1--троттлинг-рассылок--m), но шире: касается и плановых пушей из `background.py` (`send_pending_tg_notifications`, `send_pending_vk_notifications`) — они шлют пачками по 50 без пауз внутри пачки.
**Рекомендация:** общий помощник «отправить с учётом лимита» в сервисном слое, используемый и рассылкой, и пушами.

# Часть 4 — P2 Качество кода и процессы

### QA-1 · Тесты backend (pytest) · L
**Проблема.** Тестов нет вообще. Любая правка (а правок много — две фичи, аудит багов) может молча сломать права доступа или логику, и узнаем в проде.
**Варианты.**
- *(а) Узкий критичный набор:* матрица прав по ролям, парольный вход (+ lockout), привязка VK (код и пароль), доступ к файлам, генерация расписания. ~20–30 тестов закрывают 80% риска.
- *(б) Полное покрытие* всех роутеров — дорого, ROI ниже.
**Рекомендация:** (а): `pytest` + `pytest-asyncio` + `httpx.AsyncClient`, тестовая БД (SQLite в памяти для скорости или отдельная Postgres-схема). Начать с авторизации и прав.

### QA-2 · Линтеры и типизация · M
**Проблема.** Нет `ruff`, `mypy`, `eslint`, `prettier`. Стиль и типы держатся «на честном слове»; в коде остались `dict`-payload-ы и `Any`.
**Варианты.**
- *(а) `ruff` (линт+формат) + `eslint`/`prettier`* — быстрый порядок и автоисправления.
- *(б) + `mypy`/строгий `tsc`* — ловит реальные баги типов, но потребует разметки.
**Рекомендация:** (а) сразу, `mypy` — постепенно (по модулю).

### QA-3 · CI как ворота качества · M
**Проблема.** `.github/workflows/deploy.yml` при push в `main` сразу делает `rsync` + `deploy.sh`. Нет прогона тестов/линта/сборки — сломанный коммит немедленно уезжает в прод.
**Варианты.**
- *(а) Job `test` перед `deploy`* (lint + mypy + pytest + `tsc` + `vite build`); деплой только если зелёный.
- *(б) + защита ветки* (PR-обязателен, статус-чек обязателен) — процессно надёжнее.
**Рекомендация:** (а) обязательно, (б) желательно.

### QA-4 · Разбить `App.tsx` (6331 строка) · L
**Проблема.** Один файл — ~50 компонентов (Dashboard, Schedule, Normatives, Admin, Profile, Login-обвязка…). Тяжело читать, ревьюить, переиспользовать; любой конфликт слияния болезненный; невозможен нормальный code-splitting.
**Варианты.**
- *(а) По фиче:* `screens/dashboard/`, `screens/schedule/`, `screens/admin/`, общие — в `components/`, типы/хелперы — отдельно. Делать инкрементально, по одному экрану, чтобы не сломать.
- *(б) Полный рефактор за раз* — риск, лучше избегать.
**Рекомендация:** (а), параллельно с QA-5 (тесты страхуют рефактор).

### QA-5 · Frontend-тесты + e2e · M
**Варианты.** `vitest` на чистые функции (маска телефона, чётность недели, формат дат) — дёшево и ценно; Playwright e2e на вход по паролю и базовый путь участника — ловит регрессии UI.
**Рекомендация:** начать с unit на утилиты, добавить 1–2 e2e на критичные потоки.

# Часть 5 — P3 Производительность

### PERF-1 · Кэш-заголовки для статики и аватаров · S
**Проблема.** В `nginx.prod.conf` `Cache-Control: no-store` стоит на **всё** (location `/`), включая собранную статику (JS/CSS с хэшами) и `/uploads/` (аватары, файлы). Браузер перекачивает их при каждом заходе — лишний трафик и медленный старт, особенно на мобильном интернете.
**Варианты.**
- *(а) Раздельные политики:* `no-store` только для `/api/`; статика — `public, max-age=31536000, immutable` (имена с хэшами); `/uploads/` — `public, max-age=86400` (или с ETag).
- *(б) Оставить как есть* — простее, но платим скоростью.
**Рекомендация:** (а). Самая дешёвая заметная победа по скорости.

### PERF-2 · Пагинация остальных списков · S
**Проблема.** Уведомления уже пагинированы (после аудита), но аудит-лог, список пользователей, история посещаемости/сдач возвращаются целиком — растут со временем.
**Рекомендация:** единый паттерн `limit`/`offset` (как в `notifications`) + бесконечная подгрузка на фронте.

### PERF-3 · Индексы под частые фильтры · S
**Проблема.** Есть индексы на `users.telegram_id`, `users.squad_id`, добавлен `users.vk_id`. Но частые выборки идут по `event_responses(event_id,user_id)`, `attendance(event_id)`, `notifications(created_at)` — стоит проверить план запросов и докрыть.
**Рекомендация:** `EXPLAIN ANALYZE` на горячих запросах планировщика и страниц; добавить недостающие составные индексы миграцией.

### PERF-4 · Code-splitting фронтенда · M
**Проблема.** Весь `App.tsx` грузится одним бандлом, включая тяжёлые админку и отчёты с графиками — даже рядовому участнику и на экране входа.
**Варианты.**
- *(а) `React.lazy` для тяжёлых разделов* (Admin, Reports, Charts) — грузятся по требованию. Эффективно после QA-4 (разбиения файла).
- *(б) Ручной вынос графиков* в отдельный chunk.
**Рекомендация:** (а), после/вместе с QA-4.

### PERF-5 · Оптимизация изображений при загрузке · M
**Проблема.** Аватары и фото сохраняются как есть; большие файлы едят диск и трафик.
**Рекомендация:** ресайз/сжатие через Pillow при загрузке (заодно закрывает SEC-5(б)); генерация превью для видео-сдач.

# Часть 6 — P4 UX и функции

### UX-1 · PWA (установка сайта как приложения) · M
**Проблема.** Веб-вход уже есть, но сайт — обычная страница: нельзя «поставить на телефон», нет иконки, нет оффлайн-оболочки.
**Варианты.** `manifest.json` + service worker (иконки, splash, оффлайн-кэш оболочки). Vite имеет плагин `vite-plugin-pwa`.
**Рекомендация:** базовый PWA (установка + кэш статики). Полный оффлайн данных — отдельно и сложно (нужна синхронизация).

### UX-2 · Web Push для веб-сессий · L
**Проблема.** Кто заходит на сайт (не из Telegram/VK), не получает пуши.
**Варианты.** Web Push (VAPID) + service worker. Полезно, но требует разрешений браузера и серверной отправки. **Рекомендация:** после PWA, если будет спрос; для большинства хватает пушей в Telegram/VK.

### UX-3 · Сброс пароля через бота · M
**Проблема.** Забыл пароль → сейчас только переустановка в мини-аппе. Если человек на сайте и без доступа к мини-аппу — тупик.
**Варианты.**
- *(а) Через Telegram-бот:* команда `/resetpassword` → бот шлёт одноразовый код → ввод нового пароля на сайте/в боте. Канал доверенный (Telegram уже привязан).
- *(б) Через VK-бот* — аналогично, для тех, у кого привязан VK.
- *(в) Email* — у системы нет email пользователей, неприменимо.
**Рекомендация:** (а), опционально (б). Переиспользует механику кодов привязки.

### UX-4 · Доступность (a11y) · M
**Проблема.** Точечные `aria`-метки есть, но нет системного фокуса/контраста/клавиатурной навигации; много кликабельных `div`.
**Рекомендация:** видимые focus-стили, корректные роли/labels, проверка контраста, аудит axe. Важно для веб-версии.

### UX-5 · Тёмная тема · M
**Проблема.** Тема принудительно светлая — `main.tsx` снимает `data-theme`. В тёмном Telegram и вечером глаза устают.
**Варианты.** (а) следовать системной/Telegram-теме; (б) переключатель в профиле. Потребует ревизии хардкод-цветов (их много в inline-стилях и scss).
**Рекомендация:** (б) переключатель + перевод палитры на CSS-переменные (заодно облегчит будущие темы).

### UX-6 · 2FA для командиров и админов · L
**Проблема.** Доступ к данным состава защищён только паролем/Telegram. Для привилегированных ролей это мало.
**Варианты.** TOTP (authenticator-приложение) для входа на сайт под ролью командира+. **Рекомендация:** после refresh-токенов (SEC-6в); для рядовых участников избыточно.

# Часть 7 — P5 Архитектура ботов

### BOT-1 · Единый сервисный слой · L
**Проблема.** Бизнес-логика расписания/нормативов/уведомлений/ответов реализована трижды: в `bot.py` (Telegram), `vk_bot.py` (VK), роутерах API. Правило поменяли в одном месте — два других разъехались (классический источник багов).

- *(б) Оставить дубли, синхронизировать вручную* — дёшево сейчас, дорого потом.
**Рекомендация:** (а), вводить постепенно — начать с самой дублируемой логики (ответы на события, нормативы).
**Влияние:** делает реальными B-VK-1/2 (паритет VK) малой кровью.
