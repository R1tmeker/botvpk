# ТЗ: Чередующееся расписание (недели А и Б)

> Итерация: расписание. Связано с `tz-roster-birthday-export.md`, `tz-consolidated-session6.md`.
> Статус: ожидает уточнения по п.6 (глубина цикла), затем к реализации.

---

## 1. Проблема

`ScheduleTemplate` сейчас генерирует события на **каждую** неделю по выбранным дням.
Нет механизма чередования: «первый понедельник — программа А, второй — программа Б».

Одноразовые мероприятия — `ScheduleEvent` без `template_id` — уже работают корректно,
изменения их не затрагивают.

---

## 2. Концепция: Недели А и Б

Вводится понятие **цикла двух недель**: неделя А и неделя Б чередуются бесконечно.
Шаблон может работать в трёх режимах:

| `week_parity` | Поведение |
|---|---|
| `NULL` | Каждую неделю (текущее поведение, обратная совместимость) |
| `"A"` | Только в недели А |
| `"B"` | Только в недели Б |

Отсчёт ведётся от **якорной даты** — любого понедельника, назначенного «началом недели А»,
хранится в таблице `settings` под ключом `schedule_week_a_start`.

Формула определения типа недели:
```python
weeks_since_a = (monday_of_current_week - week_a_start).days // 7
parity = "A" if weeks_since_a % 2 == 0 else "B"
```

## 3. Изменения в базе данных

### 3.1 Миграция: новая колонка
```sql
ALTER TABLE schedule_templates
  ADD COLUMN week_parity VARCHAR(1) NULL
  CHECK (week_parity IN ('A', 'B'));
```

### 3.2 Новый ключ в settings
```
schedule_week_a_start = "2026-06-02"  -- ISO дата, понедельник = начало недели А
```

## 4. Backend

### 4.1 `models/schedule.py`
```python
week_parity: Mapped[str | None] = mapped_column(String(1))
```

### 4.2 `schemas/core.py`
В `ScheduleTemplateCreate`, `ScheduleTemplateRead`, `ScheduleTemplateUpdate`:
```python
week_parity: str | None = None
```

### 4.3 `routers/schedule.py` — генератор событий (`generate_template_events`)
Дополнить логику:
1. Если `template.week_parity` не `None`:
   a. Прочитать `schedule_week_a_start` из `settings` (если не задано — вернуть 400 с подсказкой)
   b. Вычислить `week_a_start = date.fromisoformat(value)`
   c. При переборе дней: если `template.week_parity` задан и текущая неделя не того типа — пропустить
2. Если `template.week_parity` равен `None` — поведение как сейчас

```python
def _week_parity_for_date(d: date, week_a_start: date) -> str:
    monday = d - timedelta(days=d.weekday())
    weeks = (monday - week_a_start).days // 7
    return "A" if weeks % 2 == 0 else "B"

# В generate_template_events:
if template.week_parity is not None:
    if _week_parity_for_date(current, week_a_start) != template.week_parity:
        current += timedelta(days=1)
        continue
```

### 4.4 `routers/admin/settings.py`
Добавить в `ALLOWED_SETTING_KEYS`:
```python

```

### 4.5 Публичный endpoint `/schedule` — опциональное обогащение ответа
Добавить в ответ текущий тип недели:
- Либо отдельный endpoint `GET /schedule/current-week-type` → `{"parity": "A"|"B"|null}`
- Либо включить в заголовок списка событий (если `schedule_week_a_start` задан)

## 6. Открытый вопрос (требует ответа перед реализацией)

**Нужен ли цикл глубже двух недель?**

- **Нет, достаточно А и Б** → текущий дизайн (этот документ).
- **Да, нужно 3+ недели** (например, А→Б→В или 4-недельный цикл) → заменяем
  `week_parity: "A"|"B"` на `cycle_position: int` (1..N) + `cycle_length: int` в шаблоне.
  Это сложнее, но гибче.

Для большинства организаций (включая ВПК) достаточно 2 недель.

## 7. Одноразовые мероприятия

**Ничего не меняется.** `ScheduleEvent` без `template_id` — это одноразовые события.
Создаются вручную через «Создать событие». Никакой генерации, никакого чередования.
Они продолжают работать ровно как сейчас.

## 7а. Одноразовое изменение конкретного занятия («исключение»)

### Проблема
Шаблон задаёт «каждый понедельник недели А — физподготовка в 16:00, спортзал».
Нужно: **на 16 июня** поставить физподготовку в 17:00, уличная площадка, тема «кросс».
Шаблон трогать нельзя — следующий понедельник должен идти по старым настройкам.

### Архитектурное решение (уже есть)
Генератор создаёт независимые строки `ScheduleEvent` в БД.
Шаблон хранится отдельно. `PATCH /schedule/events/{id}` уже позволяет изменить
заголовок, время, место, описание конкретного события без влияния на шаблон.

**Не хватает только:**
1. Поля `is_overridden: bool` в `ScheduleEvent` — маркер, что событие изменено вручную
2. Кнопки «Редактировать» в AdminView и формы редактирования
3. Визуального бейджа у изменённых событий

### Изменения в БД
```sql
ALTER TABLE schedule_events
  ADD COLUMN is_overridden BOOLEAN NOT NULL DEFAULT FALSE;
```

### Backend (`models/schedule.py`, `routers/schedule.py`, `schemas/core.py`)
- `is_overridden: Mapped[bool]` с `server_default="false"` в модели
- `is_overridden: bool = False` в `ScheduleEventRead`
- В `PATCH /schedule/events/{id}`: если `event.template_id is not None` — автоматически
  устанавливать `event.is_overridden = True` (без участия клиента)

### Frontend (AdminView — вкладка «Расписание»)
У каждого события в списке — кнопка **«Изменить»** рядом с «Удалить».
При нажатии разворачивается inline-форма с полями:
```
Название:    [_____________]   ← предзаполнено текущим значением
Начало:      [datetime-local]
Конец:       [datetime-local]
Место:       [_____________]
Описание:    [textarea     ]
```
Кнопки: «Сохранить изменение» (PATCH) / «Отмена».
После сохранения событие помечается бейджем **«изм.»** другого цвета.

### Frontend (ScheduleView — участники)
Изменённые события показываются как обычно (с новыми данными).
Опционально: маленькая пометка «время изменено» если время отличается от шаблона
(реализовывать только если не усложняет UI, иначе достаточно просто показать актуальные данные).

### Поведение при регенерации
Если администратор нажимает «Сгенерировать» на шаблоне повторно:
- Уже существующие события (определяются по `template_id` + совпадению даты) — **пропускаются**
- Создаются только события на даты, где их ещё нет
- `is_overridden = True` события не перезаписываются

Это защищает ручные изменения от случайной перезаписи при повторной генерации.

## 8. Критерии приёмки

**А/Б чередование:**
- [ ] Два шаблона «Программа А» и «Программа Б» для одного дня недели,
      сгенерированные на месяц, дают события строго в чередующемся порядке.
- [ ] Шаблон с `week_parity=NULL` работает как раньше — создаёт событие каждую неделю.
- [ ] `schedule_week_a_start` задаётся в настройках и меняет всю нумерацию без пересоздания.
- [ ] В ScheduleView участники видят, неделя А или Б сейчас.
- [ ] Одноразовые события не затронуты.
- [ ] Если `schedule_week_a_start` не задан, а шаблон с `week_parity` есть →
      генератор возвращает 400 с понятным сообщением.

**Одноразовые изменения:**
- [ ] Нажав «Изменить» на конкретном событии, можно поменять название/время/место/описание.
- [ ] Изменение касается только этого одного события, шаблон не трогается.
- [ ] Изменённое событие помечается бейджем «изм.».
- [ ] Повторная генерация шаблона не перезаписывает изменённые события.
